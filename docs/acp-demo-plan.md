# Joudo ACP Demo 计划

## 目标

使用一个独立的 demo 仓库，验证 Joudo 在真实 ACP 会话下是否能够覆盖大多数本地开发场景，包括：

- 按仓库建立和维持会话
- 安装与使用 Python 环境
- 安装与使用 Node/npm 依赖
- 编写后端和简单前端页面
- 与本地 SQLite 数据库交互
- 启动开发服务并观察运行结果
- 在需要时请求更高权限
- 通过浏览器相关工具完成页面检查与交互验证

## 建议的 demo 仓库定位

建议将 demo 仓库放在 `~/dev/demo`，并把它作为 Joudo 受信任仓库之一。

这样可以满足两个目标：

- 保持“只信任明确仓库，不信任整个 home 目录”的安全边界
- 允许我们验证 repo-scoped 会话、路径限制和越权审批流程

## 你的原计划已经覆盖的核心场景

这套流程已经覆盖了很多高价值路径，尤其包括：

### 1. 环境引导

- 调用本机已有的 pyenv
- 安装 Python 3.10
- 在仓库内创建虚拟环境
- 验证 ACP 对解释器与 shell 的调用能力

### 2. 应用搭建

- 创建 FastAPI 服务
- 提供一个简单在线页面
- 在同一个仓库中处理后端和前端资源

### 3. 依赖管理

- 检查本机 npm 现状
- 在项目内安装 npm 依赖
- 验证网络访问、安装行为和审批策略

### 4. 数据层

- 读写本地 SQLite
- 验证本地文件型数据库访问
- 验证 agent 在 repo 内的数据初始化与调试能力

### 5. 调试与交互

- 启动开发服务
- 打开浏览器
- 查看页面
- 截图或读取页面状态
- 模拟用户操作完成基本回归

### 6. 权限边界

- 读写仓库内文件
- 访问仓库外文件时触发权限请求
- 验证 repo-scoped 权限模型是否真正生效

## 这套计划还缺的关键场景

如果目标是“覆盖大部分应用场景”，还建议补上以下几类用例。

### 1. 会话生命周期

- 同一个 ACP 会话连续处理多轮 prompt
- 中断后恢复会话
- 服务重启后重新附着或重建会话

这部分很关键，因为 Joudo 的价值不只是一次性执行任务，而是持续会话。

### 2. 允许、确认、拒绝三类决策都要测

- 自动允许：repo 内读写、受控测试、受控启动命令
- 需要确认：安装依赖、联网请求、访问仓库外路径
- 自动拒绝：危险 shell、广泛破坏性命令、越界路径写入

如果只测“能做什么”，策略系统很容易看起来可用，但实际上没有验证真正的边界。

### 3. 失败恢复

- 依赖安装失败后重试
- 端口占用时调整运行配置
- SQLite 文件不存在时自动初始化
- Python 或 npm 命令失败后根据报错修复

真实开发流程里，失败恢复通常比首次成功更常见。

### 4. Git 工作流

- 查看状态
- 查看 diff
- 检查改动文件

这不一定要一开始就测提交和 push，但至少要验证 ACP 能把代码改动回写成一个可审视的工作区状态。

### 5. 审计与摘要

- 执行过哪些命令
- 修改了哪些文件
- 跑了哪些检查
- 哪些动作被拒绝或被确认

这部分直接关系到 Joudo 网页端的摘要价值。

### 6. 网络与代理

- npm 或 Python 包安装需要联网
- 当网络失败时，是否按规则切换代理后重试

这是当前机器上的现实约束，不应只当成环境偶发问题。

## 建议的分阶段执行顺序

### Phase 1: 仓库与会话基线

- 创建 `~/dev/demo`
- 初始化为一个最小可工作的仓库目录
- 为它准备 Joudo 策略文件
- 用 Joudo bridge 启动真实 ACP 会话
- 验证会话创建、prompt 转发、事件回流、摘要展示

### Phase 2: Python 环境与 FastAPI

- 检查 pyenv 是否存在 Python 3.10
- 必要时安装 Python 3.10
- 在 demo 内创建虚拟环境
- 搭一个最小 FastAPI 服务
- 让服务返回 HTML 页面
- 用 SQLite 完成一个最小读写闭环

### Phase 3: npm 与前端资源

- 检查 npm/node 可用性
- 在 demo 仓库内安装 npm 依赖
- 增加一个最小前端页面或静态资源构建链
- 验证 Joudo 对安装、运行和调试前端依赖的策略行为

### Phase 4: 浏览器调试

- 启动服务
- 打开页面
- 读取页面状态
- 截图
- 模拟点击、输入、提交流程

这一阶段能验证 Joudo 是否能承担“移动端远程督导一个本地 agent”这件事。

### Phase 5: 权限边界与拒绝路径

- 访问 repo 内文件
- 请求访问 repo 外文件
- 对越界路径写入进行确认或拒绝
- 尝试一个应被拒绝的危险动作

这一阶段是策略系统的关键验收，不应省略。

### Phase 6: 失败恢复与总结

- 人为制造一个安装或运行错误
- 让 ACP 会话根据错误恢复
- 检查最终摘要是否能准确说明执行过程、结果和阻塞点

## 建议的验收标准

如果下面这些点都能通过，基本可以说明 Joudo 已覆盖大量真实场景：

- 能在一个明确仓库根目录上稳定运行真实 ACP 会话
- 能跨多轮 prompt 持续维护上下文
- 能完成 Python 和 Node 双栈环境操作
- 能完成后端、页面、数据库三层联调
- 能通过浏览器工具观察和验证页面行为
- 能对 repo 外访问触发正确审批
- 能对危险动作拒绝或要求确认
- 能把执行过程收敛成面向用户的结构化摘要

你的原计划是对的，而且已经覆盖了大部分高价值主路径。

目标是让网页成为可靠的控制面板，而不是只显示零散事件。
