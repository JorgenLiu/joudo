# Joudo 独立审核报告 — 2026-03-22

## 一、整体评价

Joudo 在"Local-first Copilot CLI control plane"这个方向上已经走得比大多数同类尝试更远。核心价值链——repo-scoped policy → 权限决策 → 审批闭环 → 持久化治理——已经真实打通，不是 demo。这是一个有真正产品判断力的项目。

但诚实地说：**当前状态是一个高质量的内部工具原型，而不是一个可以向外交付的产品。** 差距不在功能缺失——功能覆盖面已经超出 MVP 通常需要的范围——而在运行时健壮性和安全边界还停留在"开发者自己用可以接受"的水平。

## 二、最关键的运行时缺陷

### 2.1 `ensureClient()` 无并发保护 — 真实 bug

**位置**：`apps/bridge/src/state/session-runtime.ts` 第 103 行

**问题**：两个并发调用同时通过 `if (clientStartPromise)` 检查时，会创建两个 `CopilotClient` 并启动两次。第一个 promise 被覆盖后丢失引用。`recoverHistoricalSession` 和 `refreshAuthState` 完全可以同时触发 `ensureClient()`。

**修复方案**：

```typescript
// 修复前
async function ensureClient() {
  if (deps.clientRuntimeRef.clientStartPromise) {
    await deps.clientRuntimeRef.clientStartPromise;
    // ...
  }
  const nextClient = new CopilotClient({ ... });
  deps.clientRuntimeRef.client = nextClient;
  deps.clientRuntimeRef.clientStartPromise = nextClient.start();
  // ...
}

// 修复后：先占位 promise，再创建 client
async function ensureClient() {
  if (deps.clientRuntimeRef.clientStartPromise) {
    await deps.clientRuntimeRef.clientStartPromise;
    if (!deps.clientRuntimeRef.client) {
      throw new Error("Copilot client 初始化失败。");
    }
    return deps.clientRuntimeRef.client;
  }

  // 先设置 promise 占位，阻止并发进入
  let resolveStart!: () => void;
  let rejectStart!: (err: unknown) => void;
  deps.clientRuntimeRef.clientStartPromise = new Promise<void>((res, rej) => {
    resolveStart = res;
    rejectStart = rej;
  });

  try {
    const nextClient =
      deps.createClient?.(deps.currentContext()?.repo.rootPath ?? process.cwd()) ??
      new CopilotClient({
        cwd: deps.currentContext()?.repo.rootPath ?? process.cwd(),
        logLevel: "error",
      });
    deps.clientRuntimeRef.client = nextClient;
    await nextClient.start();
    resolveStart();
    return nextClient;
  } catch (error) {
    deps.clientRuntimeRef.clientStartPromise = null;
    deps.clientRuntimeRef.client = null;
    rejectStart(error);
    throw error;
  }
}
```

**验证方法**：在 session-runtime 新测试文件中写一个并发测试：

```typescript
// 两次并发调用应只创建一个 client
const [a, b] = await Promise.all([ensureClient(), ensureClient()]);
assert.strictEqual(a, b);
assert.strictEqual(createClientCallCount, 1);
```

### 2.2 Repo 切换不清理运行中的 prompt 和 pending approvals

**位置**：`apps/bridge/src/mvp-state.ts` 第 540 行 `selectRepo()`

**问题**：`selectRepo` 只切换 `currentRepoId`，不碰旧 repo 的 `lifecycle.activePrompt` 和 `pendingApprovals`。如果用户在 prompt 运行中切换 repo：

- 旧 repo 的 `handlePermissionRequest` 仍会被回调
- approval 解析找不到对应 repo（`approvalRepoIndex` 的 repoId 已经不是 current）
- Copilot SDK 那边的 permission callback 挂起，永远不 resolve

**修复方案**：在 `selectRepo` 中检测旧 repo 是否有运行中的 prompt 或 pending approvals：

```typescript
selectRepo(repoId: string) {
  // ... 现有 mutationInFlight 检查 ...

  const oldContext = currentContext();
  if (oldContext && oldContext.repo.id !== repoId) {
    // 方案 A（推荐）：如果有活跃 prompt，拒绝切换
    if (oldContext.lifecycle.activePrompt) {
      throw new JoudoError("validation", "当前仓库有正在运行的任务，请等待完成或取消后再切换。", {
        statusCode: 409,
        nextAction: "等待当前 prompt 执行完毕，或先取消当前任务。",
        retryable: true,
      });
    }

    // 方案 B（可选）：强制清理旧 context
    // await disconnectRepoSession(oldContext);
    // oldContext.approvalState.pendingApprovals.clear();
    // oldContext.lifecycle.activePrompt = null;
  }

  currentRepoId = repoId;
  // ... 后续逻辑 ...
}
```

推荐方案 A：拒绝比静默丢弃更安全，用户会收到明确提示。

**验证方法**：

```typescript
// 测试：prompt 运行中切换 repo 应抛出 409
state.submitPrompt("do something");
assert.throws(() => state.selectRepo("other-repo"), /409/);
```

### 2.3 XSS：MarkdownBody link href 未做协议验证

**位置**：`apps/web/src/components/MarkdownBody.tsx` 第 80 行

**问题**：markdown 解析出的 URL 直接放进 `<a href={token.href}>`。如果 Copilot agent 返回的 markdown 包含 `[click](javascript:alert(1))`，浏览器会执行。

**修复方案**：

```typescript
// 在 renderInlineTokens 的 link case 中
case "link": {
  const isSafe = /^https?:\/\//i.test(token.href) || token.href.startsWith("/") || token.href.startsWith("#");
  const href = isSafe ? token.href : "#";
  return <a key={key} href={href} target="_blank" rel="noopener noreferrer" className="mdLink">{token.text}</a>;
}
```

**验证方法**：在 MarkdownBody 测试中增加：

```typescript
it("strips javascript: protocol from links", () => {
  const { container } = render(<MarkdownBody text="[xss](javascript:alert(1))" />);
  const link = container.querySelector("a");
  assert.strictEqual(link?.getAttribute("href"), "#");
});
```

## 五、POST Route Runtime Validation

### 问题

当前所有 POST 路由只有 TypeScript 泛型约束（`Body: PromptSubmission` 等），运行时不做校验。恶意或格式错误的请求体会直接穿透到业务逻辑。

### 推荐方案：Fastify JSON Schema

不引入新依赖（Fastify 内置 Ajv），在路由定义上补 schema：

```typescript
app.post<{ Body: PromptSubmission }>("/api/prompt", {
  schema: {
    body: {
      type: "object",
      required: ["prompt"],
      properties: {
        prompt: { type: "string", minLength: 1, maxLength: 10000 },
      },
      additionalProperties: false,
    },
  },
}, async (request) => {
  return state.submitPrompt(request.body.prompt);
});
```

### 需要加 schema 的路由清单

| 路由 | 必填字段 | 约束 |
|------|----------|------|
| `POST /api/prompt` | `prompt: string` | 1–10000 字符 |
| `POST /api/session/select` | `repoId: string` | 非空 |
| `POST /api/session/model` | `model: string` | 非空 |
| `POST /api/session/recover` | `joudoSessionId: string` | 非空 |
| `POST /api/session/resume` | `joudoSessionId: string` | 非空 |
| `POST /api/approval` | `approvalId: string`, `decision: string` | decision ∈ enum |
| `POST /api/repo/instruction` | `userNotes: string` | maxLength: 50000 |
| `POST /api/repo/policy/rule/delete` | `field: string`, `value: string` | 非空 |
| `POST /api/auth/totp` | `code: string` | 6 位数字 |
| `POST /api/session/rollback` | — | 无 body |

### 操作步骤

1. 在 `apps/bridge/src/index.ts` 中给上述每个 POST 路由添加 `schema.body`
2. Fastify 自动返回 400 + 错误明细，无需手动 catch
3. 可以考虑把 schema 集中定义在 `apps/bridge/src/route-schemas.ts` 中，但第一版直接 inline 也可以

## 六、CI/CD 管线 — 使用 GitHub Actions

### 当前状态

项目仓库在 GitHub 上，但没有任何 CI 配置（`.github/` 下只有 `copilot-instructions.md`）。

### 推荐方案

直接使用 **GitHub Actions**。理由：

1. 代码已在 GitHub 托管，零配置集成
2. Private repo 每月有 2,000 分钟免费 Actions 额度（足够）
3. pnpm + Node.js 生态在 Actions 上成熟度极高
4. macOS runner 可用（Copilot CLI 依赖 macOS 的测试可以 macOS runner 跑）

### 第一阶段：基础 CI（立即可做）

创建 `.github/workflows/ci.yml`：

```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: pnpm/action-setup@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: pnpm

      - run: pnpm install --frozen-lockfile

      - name: Typecheck
        run: pnpm typecheck

      - name: Bridge tests
        run: pnpm --filter @joudo/bridge test

      - name: Web tests
        run: pnpm --filter @joudo/web test

      - name: Web build
        run: pnpm --filter @joudo/web build
```

**注意事项**：

- `pnpm/action-setup@v4` 会自动读取 `package.json` 中的 `packageManager` 字段
- Bridge 测试不需要真实 Copilot CLI，当前全靠 mock/stub
- Web build 验证确保 Vite 打包不会在 import 解析上报错

### 第二阶段：Bridge Build + Artifact

当前 `@joudo/bridge` 没有 `build` script。需要先补：

```jsonc
// apps/bridge/package.json scripts 新增
"build": "tsc -p tsconfig.build.json"
```

加一个 `tsconfig.build.json`：

```jsonc
{
  "extends": "./tsconfig.json",
  "compilerOptions": {
    "outDir": "dist",
    "declaration": true,
    "sourceMap": true
  },
  "exclude": ["src/**/*.test.ts"]
}
```

然后 CI 中加：

```yaml
      - name: Bridge build
        run: pnpm --filter @joudo/bridge build
```

### 第三阶段：Release Artifact（packaging 时再加）

等 packaging 就绪后，CI 可以做：

- `vite build` 产出 Web 静态资源 → 拷贝到 bridge dist 的 `public/` 目录
- bridge 用 `@fastify/static` 托管打包后的 Web 资源
- CI 产出一个可运行的 `dist/` 目录作为 artifact

## 七、是否需要桌面 App 界面

### 结论：**当前不需要独立桌面 App，但需要一个"启动器"脚本。**

### 分析

Joudo 的核心价值是 **bridge + Web UI**。桌面 App 外壳只是一个 launcher，不参与任何业务逻辑。引入 Electron 或 Tauri 会带来：

- 构建复杂度翻倍（macOS 签名、universal binary、自动更新）
- 当前开发机是 macOS Ventura 旧硬件，Electron 构建工具链可能不兼容
- 维护表面积增大，对只有一个开发者的项目不划算

### 推荐方案：分三步解决"启动体验"问题

#### 第一步（现在就做）：`joudo` CLI 入口脚本

一个 shell 脚本，解决"用户不需要理解 pnpm 命令"的问题：

```bash
#!/usr/bin/env bash
# scripts/joudo-start.sh

set -euo pipefail

JOUDO_ROOT="$(cd "$(dirname "$0")/.." && pwd)"
cd "$JOUDO_ROOT"

echo "🔧 Starting Joudo bridge..."
# 编译并启动 bridge（前台），bridge 会同时托管 Web 静态资源
node apps/bridge/dist/index.js &
BRIDGE_PID=$!

# 等 bridge 就绪
for i in {1..30}; do
  if curl -sf http://localhost:8787/health > /dev/null 2>&1; then
    break
  fi
  sleep 0.5
done

# 自动打开浏览器
LOCAL_IP=$(ipconfig getifaddr en0 2>/dev/null || echo "localhost")
echo "✅ Joudo is running at http://${LOCAL_IP}:8787"
open "http://localhost:8787"

# 前台等待
wait $BRIDGE_PID
```

#### 第二步（packaging 阶段）：macOS LaunchAgent

让 Joudo bridge 开机自启或按需启动：

```xml
<!-- ~/Library/LaunchAgents/com.joudo.bridge.plist -->
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>Label</key>
  <string>com.joudo.bridge</string>
  <key>ProgramArguments</key>
  <array>
    <string>/usr/local/bin/node</string>
    <string>/path/to/joudo/apps/bridge/dist/index.js</string>
  </array>
  <key>RunAtLoad</key>
  <false/>
  <key>KeepAlive</key>
  <false/>
</dict>
</plist>
```

用户通过 `launchctl start com.joudo.bridge` 启动，或点一个 `.command` 脚本双击启动。

#### 第三步（如果确实需要桌面壳）：考虑 Tauri 而非 Electron

如果日后确实需要系统托盘图标、全局快捷键等原生能力，Tauri v2 比 Electron 更轻量：

- 单 binary，不带 Chromium
- Rust 后端 + WebView 前端
- macOS 打包产物 ~10MB vs Electron ~150MB
- 但学习曲线和调试复杂度更高

**当前建议：不要在 packaging-release-plan.md 中承诺桌面 App 时间线。** 先把 CLI 启动器 + bridge 静态托管做好，这就足以让用户"双击就能用"。

## 八、Repo Policy 初始化体验

### 当前问题

用户要在一个目录上使用 Joudo 时，需要手动创建 `.github/joudo-policy.yml`。没有引导流程。

### 推荐方案：bridge 侧 `init` 命令 + Web UI 引导

#### 8.1 Bridge：`POST /api/repo/init-policy`

当 Web UI 检测到选中的 repo 没有 policy 文件时，展示引导卡片，用户点击后调用此接口。

```typescript
app.post<{ Body: { repoId: string } }>("/api/repo/init-policy", async (request) => {
  return state.initRepoPolicy(request.body.repoId);
});
```

Bridge 侧实现：

```typescript
async function initRepoPolicy(repoId: string) {
  const context = repoContexts.get(repoId);
  if (!context) throw new JoudoError("validation", "仓库不存在。", { statusCode: 404 });
  if (context.policy.state !== "not-found") {
    throw new JoudoError("validation", "Policy 文件已存在。", { statusCode: 409 });
  }

  const policyPath = path.join(context.repo.rootPath, ".github", "joudo-policy.yml");
  await fs.mkdir(path.dirname(policyPath), { recursive: true });
  await fs.writeFile(policyPath, DEFAULT_POLICY_TEMPLATE, "utf8");
  refreshRepoPolicy(context);
  return snapshot();
}
```

默认模板（参考 `docs/examples/joudo-policy.recommended.yml` 已有的）：

```yaml
# 需要确认的 shell 命令
confirm_shell: []

# 禁止的 shell 命令
deny_shell: []

# 允许写入的路径

# 允许读取的路径

```

## 九、TOTP 绑定体验

### 当前问题

TOTP 绑定依赖终端 QR 码（`qrcode-terminal` 包在 bridge 启动时打印到 stdout）。在打包后的产品中，用户可能看不到终端。

### 推荐方案：Web UI 内完成 TOTP 绑定

#### 9.1 新增 `GET /api/auth/totp/setup` 接口

首次设置时，返回 TOTP secret 的 QR 图片和手动输入 key：

```typescript
app.get("/api/auth/totp/setup", async (request, reply) => {
  // 只在尚未完成首次绑定时允许访问
  // 或者只在 LAN 内、首次启动时允许
  if (!isNewTotpSecret) {
    void reply.status(403).send({ error: "TOTP already configured." });
    return;
  }

  const otpauthUrl = `otpauth://totp/Joudo?secret=${base32Encode(totpSecret)}&issuer=Joudo`;
  // 用 JS 库生成 QR data URL（替代 qrcode-terminal）
  const qrDataUrl = await generateQrDataUrl(otpauthUrl);

  return {
    qrDataUrl,        // data:image/png;base64,...
    manualKey: base32Encode(totpSecret),  // 手动输入用
    otpauthUrl,        // 直接点击跳转验证器 app
  };
});
```

#### 9.2 Web UI：TotpSetup 组件

当用户首次访问且未绑定 TOTP 时，展示：

```
🔐 首次设置 — 绑定验证器

1. 打开手机上的验证器应用（Google Authenticator / 1Password / Authy）
2. 扫描下方二维码，或手动输入密钥

[QR Code Image]

手动输入密钥：JBSW Y3DP EHPK 3PXP

3. 输入验证器显示的 6 位数字完成绑定：
[______] [确认绑定]
```

#### 9.3 安全考量

- `GET /api/auth/totp/setup` **必须只在首次绑定时可用**。一旦 TOTP 已配置，此端点返回 403。
- 可以加一个额外保护：只在 bridge 启动后 5 分钟内允许访问 setup 端点。
- QR 图片生成建议用 `qrcode` npm 包（纯 JS，无 native 依赖），替代当前的 `qrcode-terminal`：

## 十一、与前一轮评估的差异

| 前一轮评估观点 | 本轮意见 |
|---------------|---------|
| Auth/transport hardening 是最重要的下一步 | **不同意**。LAN-first 场景下，引擎可靠性比 auth 更紧迫 |
| 3 阶段打包路线（单服务 → VPN → Internet） | **方向同意，但缺少前置步骤**：先要有 CI + build + static hosting |
| TOTP 终端依赖是阻塞项 | **同意**，但解决方案应该是 Web UI 内绑定而非桌面 App |
| `withCopilotRetry` 只是 paper fix | **同意**，建议直接删除而非强行接入 |
| 需要桌面 App | **当前不需要**。CLI 启动器 + 浏览器足够，Tauri 可以作为后续选项评估 |
