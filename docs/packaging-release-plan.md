# Joudo 打包与发布计划

## 当前结论

Joudo 当前还不处于正式发布阶段。

当前最重要的事情不是“尽快打包”，而是先把：

- bridge
- Web UI
- policy
- 恢复与 rollback

这些核心产品面收敛稳定。

## 当前交付形态

当前形态是开发态组合：

- 一个本地运行的 bridge
- 一个 Vite 驱动的 Web UI
- 一套 repo-scoped 持久化与策略文件

这意味着：

- 适合开发和局域网演示
- 不适合直接当成最终用户安装体验

## 打包目标

后续 packaging 的合理方向应是：

- 保留本地优先
- 保留 repo-scoped 安全边界
- 不把 Copilot CLI 权限误扩大成系统级宽权限壳
- 让用户不需要手动分别启动 bridge 和 web

## 当前推荐分阶段路线

### 第一阶段：开发态稳定

目标：

- 保持 `pnpm dev` 工作流稳定
- 保持类型检查、bridge 测试、web 测试稳定
- 继续收敛文档与产品面

### 第二阶段：单机可运行收敛

目标：

- 让 bridge 与 Web UI 更容易作为一个“本地应用组合”启动
- 尽量减少用户对开发命令的理解负担

### 第三阶段：正式安装形态评估

目标：

- 再决定是否封装为桌面应用
- 再决定是否需要后台服务或系统登录项

## 当前不建议提前做的事

- 在产品面未稳定前过早做安装器
- 为了打包方便而削弱 repo-scoped policy 与审批边界
- 依赖云中继来回避本地产品问题

## 当前 packaging 风险

### 安全边界不能在打包阶段被抹平

Joudo 的价值之一就是 repo-scoped control plane。如果打包时把它收敛成“一个宽权限本地代理”，就会丢失产品核心。

### bridge 与 CLI 仍然有本地运行前提

当前产品仍以宿主机可运行 Copilot CLI 为前提，打包不能掩盖这一点。

### 当前恢复与 rollback 语义仍在收敛

在这些关键能力还在变化时，过早固化安装形态会放大维护成本。

## 当前阶段建议

先把 Joudo 当成一个正在收敛中的本地产品，而不是一个已经准备发布的安装包项目。
